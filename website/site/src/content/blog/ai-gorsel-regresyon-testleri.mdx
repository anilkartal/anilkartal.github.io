Klasik snapshot testleri güçlüdür ama tek piksel kayması bile tüm suite'i kırmaya yeter. Bir tasarım tokeni güncellemesinin ardından onlarca sayfada elle screenshot kabul etmek, "bu testi kim onayladı?" sorusunu anlamsızlaştırır. AI tabanlı görsel regresyon araçları bu denklemi değiştiriyor: pikseli değil, anlamı karşılaştırıyor.

## Piksel Karşılaştırması Nerede Yetersiz Kalır

Playwright'ın built-in `toHaveScreenshot()` methodu deterministik ve hızlıdır. Ancak şu senaryolarda sürtünme yaratır:

- Font anti-aliasing farklılıkları (OS veya render engine bazında)
- Dinamik içerik (tarih, kullanıcı adı, banner metni)
- Küçük CSS değişiklikleri sonrası tüm sayfada kademeli kayma

Bu durumlarda geliştirici zamanının önemli bir kısmı "neden kırıldı?" değil "sahte mi gerçek mi?" sorusunu yanıtlamaya gider.

## AI Yaklaşımı Nasıl Çalışır

Modern AI görsel test araçları (Argos, Percy, Applitools Eyes) görselleri piksel gridi olarak değil, semantic bölgeler halinde analiz eder. "Bu buton hizası kaymış mı?" sorusunu, insan gözünün değerlendireceği gibi yanıtlar.

```typescript
// playwright.config.ts — Argos entegrasyonu
import { defineConfig } from '@playwright/test';

export default defineConfig({
  reporter: [
    ['list'],
    ['@argos-ci/playwright/reporter'],
  ],
});
```

```typescript
// tests/visual/home.spec.ts
import { test } from '@playwright/test';
import { argosScreenshot } from '@argos-ci/playwright';

test('anasayfa görsel tutarlılık', async ({ page }) => {
  await page.goto('/');
  await page.waitForLoadState('networkidle');
  
  // Dinamik içeriği gizle
  await page.evaluate(() => {
    document.querySelectorAll('[data-testid="timestamp"]')
      .forEach(el => (el as HTMLElement).style.visibility = 'hidden');
  });

  await argosScreenshot(page, 'home-page');
});
```

Argos, görüntüyü CI'a yükler; AI modeli baseline ile karşılaştırır ve yalnızca anlamlı değişiklikleri işaretler. Font rendering veya sub-pixel farklılıkları gürültü olarak süzülür.

## Ne Zaman Mantıklı, Ne Zaman Değil

**Mantıklı:**
- Tasarım sistemi token'larının tutarlılığını doğrulamak
- Marketing sayfaları gibi görsel kararlılığın kritik olduğu alanlar
- Multi-browser cross-platform testler

**Overkill:**
- Tek bir bileşenin unit test düzeyinde görsel testi
- Her sprint'te şekil değiştiren prototip aşamasındaki UI'lar
- API veya iş mantığı odaklı test suite'leri

## Mevcut Playwright Suite'e Entegrasyon Stratejisi

Görsel testleri mevcut fonksiyonel testlerden ayırmak yönetimi kolaylaştırır. Playwright'ın `projects` özelliği bunu temiz bir şekilde sağlar:

```typescript
export default defineConfig({
  projects: [
    { name: 'functional', testDir: './tests/functional' },
    {
      name: 'visual',
      testDir: './tests/visual',
      // Visual testler yalnızca main branch'te veya release öncesi çalışsın
      grep: process.env.RUN_VISUAL ? undefined : /^$/, 
    },
  ],
});
```

Bu yaklaşımda feature branch'lerde sadece fonksiyonel testler çalışır, görsel karşılaştırmalar release pipeline'ına rezerve edilir. CI süreleri kontrol altında kalır, false positive'ler geliştiricileri bunaltmaz.

AI görsel test araçları mükemmel değil; özellikle animasyon veya scroll pozisyonu hassasiyeti gereken durumlarda ek yapılandırma gerektirir. Ama doğru bağlamda, görsel regresyon tespitini elle yapılabilir olmaktan çıkarıp sistematik bir kalite katmanına dönüştürüyor.
